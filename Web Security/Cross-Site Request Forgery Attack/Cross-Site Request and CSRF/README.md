# Cross-Site Requests and CSRF

## What is a Cross-Site Request?

When browsing the web, we constantly interact with multiple websites. A **cross-site request** occurs when a webpage from one website sends a request to a different website.

![Cross-Site Request](https://github.com/user-attachments/assets/eeb502eb-910b-4175-a42b-1422143806a4)

For example:

- If you are on **Facebook** and that page sends a request to Facebook, it is considered a **same-site request** because both the source and destination belong to the same website.
- Likewise, if you are on another website (e.g., Orange) and it sends a request to itself, that is also a **same-site request**.
- However, if a webpage from **Orange** sends a request to **Facebook**, it becomes a **cross-site request** because the request originates from a different website.

Cross-site requests are a normal and essential part of how the web works. For example, clicking a Facebook link from another website generates a cross-site request, allowing users to navigate between websites seamlessly.

### Why Is This Important?

By design, browsers automatically attach any cookies associated with the destination website whenever they send a request. This behavior occurs regardless of whether the request is same-site or cross-site.

This includes **session cookies**, which identify an authenticated user.

As a result, if a user is logged into Facebook and visits a malicious website, any request that website causes the browser to send to Facebook will automatically include the user's Facebook session cookie.

This browser behavior is what makes **Cross-Site Request Forgery (CSRF)** attacks possible.

# Session Hijacking Using CSRF

CSRF can be thought of as a form of session hijacking. Unlike traditional session hijacking, however, the attacker does **not** need to steal or guess the victim's session identifier.

## Traditional TCP Session Hijacking

![TCP Session Hijacking](https://github.com/user-attachments/assets/775778df-69a1-4840-81a5-5721892ee7ec)

In TCP session hijacking:

- Two parties communicate using TCP.
- Every TCP session is identified using **sequence numbers**.
- If an attacker can predict or observe the sequence number, they can inject malicious packets into the connection and hijack the session.

Although sequence numbers are not secret, they are difficult enough to predict that they provide protection against attackers.

## CSRF Session Hijacking

![CSRF Session Hijacking](https://github.com/user-attachments/assets/458b99dc-e93b-430f-890e-8d969607925e)

Web applications use **session IDs** instead of TCP sequence numbers.

These session IDs are:

- Randomly generated
- Difficult to guess
- Usually protected by HTTPS, making them difficult to intercept

Instead of stealing the session ID, a CSRF attacker takes advantage of the browser itself.

The attack works as follows:

1. The attacker convinces the victim to visit a malicious webpage.
2. The malicious page executes code inside the victim's browser.
3. That code sends a request to the target website.
4. The browser automatically includes the victim's authenticated session cookie.
5. The target website processes the request as if it came directly from the legitimate user.

The attacker never learns the victim's session ID. Instead, they exploit the browser's built-in behavior of automatically attaching session cookies to requests.

If the target website cannot distinguish between legitimate requests and requests generated by a malicious webpage, the attack succeeds.

# CSRF Attacks on IoT Devices

![CSRF on IoT Devices](https://github.com/user-attachments/assets/d6d70b69-f00d-4a52-ac26-d006b12fd386)

Internet of Things (IoT) devices are often vulnerable to CSRF because they commonly rely on network isolation instead of implementing proper web security protections.

A typical home network works as follows:

- The IoT device hosts a small web server used for configuration.
- The device is assigned a **private IP address**, making it inaccessible directly from the Internet.
- A home router or firewall blocks external access.
- Users inside the home network can freely access the device through its web interface.

This design assumes that anyone inside the local network is trustworthy.

However, this assumption fails if an attacker can execute code inside a trusted user's browser.

For example:

1. A user visits a malicious website.
2. The attacker's webpage runs JavaScript or automatically submits an HTTP request.
3. The browser sends requests to the IoT device on the local network.
4. Since the request originates from inside the trusted network, the IoT device accepts it.

Even though the attacker is outside the home network, they can indirectly interact with the IoT device through the victim's browser.

Many IoT devices lack CSRF protections such as CSRF tokens or Origin/Referer validation, making them particularly susceptible to these attacks.
